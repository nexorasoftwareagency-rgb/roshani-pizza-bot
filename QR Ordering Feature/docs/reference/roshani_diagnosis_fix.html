The Real Root Cause — Not What ChatGPT Said

ChatGPT diagnosed "invalid table markup / foster parenting" around inventoryTableBody. That is wrong. The actual HTML structure is <div class="table-responsive"><div id="inventoryTableBody"></div></div> — a div inside a div, perfectly valid HTML. No foster parenting occurs.

The actual bugs causing offsPar=null · sH=0 · oH=0 are:

Bug 1 (Primary): promotions.js calls its internal _attachCampaignListener() on page load (Firebase onValue fires immediately with cached data). The layout measurement code inside runs before the user has ever opened the Promotions tab. At that moment, tab-promotions still has class="hidden" = display:none. Any DOM measurement on a display:none element returns 0 — always.

Bug 2 (Secondary): ui.js switchTab() runs an await Promise.allSettled(cleanupTasks) block that imports and calls cleanup on every other module before running the active tab's data load. On slow devices, this means the tab is visible (hidden removed) but the module that renders its content hasn't executed yet — leaving the tab showing blank/height-0 content for 200–800ms.

Bug 3 (Auth): role: undefined in Firebase — admin profile missing role field. Causes silent permission failures on protected operations.

🔴 Root Cause 1 — Primary Bug

RC-01 promotions.js runs layout measurements on a display:none tab — always gets 0 Critical promotions.js ▾

Exact mechanism: When the page loads, auth.js calls onAuthStateChanged → Firebase confirms the admin → some init path triggers _attachCampaignListener() inside promotions.js → Firebase onValue fires immediately with cached data → the callback runs layout measurement code (the debug block with offsPar, sH, oH) via a setTimeout(fn, 1000).

At that point, #tab-promotions still has class="tab-content hidden". In your CSS, .hidden { display: none !important }. Any element with display:none returns:
• el.offsetParent = null ← matches "offsPar=null"
• el.scrollHeight = 0 ← matches "sH=0"
• el.offsetHeight = 0 ← matches "oH=0"
• el.isConnected = true ← matches "isConn=true" (it IS in DOM, just hidden)
• el.parentElement = div.table-responsive ← matches "parent=div.table-responsive"
• document.querySelectorAll('.main').length = 1 ← matches "numMain=1"

Every single value in the debug log is explained by one fact: the measurement ran while the tab had display:none.

The fix is a visibility guard. Before any layout measurement or Tabulator initialization, check that the container is actually visible. If it's not, defer initialization to when the tab is opened.

promotions.js — runs while hidden
// Somewhere in promotions.js // (called from Firebase onValue callback): function _attachCampaignListener() { // ...Firebase listener... onValue(ref, (snap) => { // This runs on PAGE LOAD // tab is still display:none! const container = document .querySelector('.table-responsive'); // Debug block that produces the log: setTimeout(() => { console.log({ offsPar: container?.offsetParent, sH: container?.scrollHeight, // = 0 oH: container?.offsetHeight, // = 0 parent: container?.parentElement, numMain: document .querySelectorAll('.main').length, isConn: container?.isConnected }); }, 1000); // Then tries to render into // a display:none container... }); }
Fixed — visibility guard
// In promotions.js — add this utility: function isTabVisible(tabId) { const tab = document.getElementById( `tab-${tabId}`); if (!tab) return false; return !tab.classList.contains('hidden') && tab.offsetParent !== null; } // Wrap any layout measurement or // Tabulator init with this guard: function _attachCampaignListener() { onValue(ref, (snap) => { // Store the data in state _campaignData = snap.val() || {}; // Only render if tab is visible if (isTabVisible('promotions')) { _renderCampaigns(_campaignData); } // Otherwise: loadPromotions() will // call _renderCampaigns() when the // admin actually opens the tab }); } // loadPromotions is already called by // switchTab when user opens the tab. // Just make sure it always renders // from the cached _campaignData: export function loadPromotions() { if (_campaignData) { _renderCampaigns(_campaignData); } // Also attach listener if not attached if (!_listenerAttached) { _attachCampaignListener(); _listenerAttached = true; } }
  • The tab measuring bug is eliminated — no more height=0 / offsPar=null debug output
  • Data still loads in the background — but rendering only happens when the tab is visible
  • Same pattern should be applied to Discounts, Feedback, and Walkin if they have the same issue
  • The same isTabVisible() utility works for all tabs

🟠 Root Cause 2 — Sequential cleanup delay in switchTab

RC-02 switchTab awaits cleanup of ALL other modules before loading the target tab's data High ui.js ▾

In ui.js switchTab(), after removing .hidden from the target tab, the code runs:

const cleanupTasks = [...many modules...];
await Promise.allSettled(cleanupTasks);

This await block dynamically imports and executes cleanup functions from 9 different feature modules before the data load for the active tab begins. On first tab visit, each of those imports is a network fetch. On slow mobile connections (the exact scenario you're testing), this can take 200–800ms.

During this wait, the tab is visible (no .hidden) but completely empty — the content hasn't rendered yet. If any of these cleanup promises hang (e.g., tracker.js tries to stop a location listener before it was started), the entire chain stalls. The console log [SWITCH] cleanup done for ${tabId} fires only after ALL of them resolve.

Additionally, the message handler took 493ms log you shared maps exactly to this cleanup block — the click handler dispatches a tab switch, the await cleanup takes 493ms, during which the UI appears frozen.

Run cleanup in the background — don't await it before loading the target tab. Show the tab content immediately.

ui.js — awaits cleanup first
// Current: content waits for cleanup target.classList.remove('hidden'); // ❌ AWAITS all cleanup before // loading the tab the user wants const cleanupTasks = []; if (tabId !== 'catalog') cleanupTasks.push( mod('catalog').then( m => m.cleanupCatalog?.())); // ... 9 more cleanup tasks ... await Promise.allSettled(cleanupTasks); // User sees empty tab for 200-800ms console.log('[SWITCH] cleanup done'); // THEN loads active tab data switch(tabId) { case 'promotions': const { loadPromotions } = await mod('promotions'); loadPromotions(); break; }
Fixed — fire-and-forget cleanup
target.classList.remove('hidden'); // ✅ Load the target tab FIRST // Don't wait for cleanup try { switch(tabId) { case 'promotions': { const { loadPromotions } = await mod('promotions'); loadPromotions(); break; } // ... all other cases ... } } catch (loadErr) { logger.error('TAB', `Load error: ${loadErr.message}`); } // THEN cleanup other tabs // in the background (no await) const cleanupTasks = []; if (tabId !== 'catalog') cleanupTasks.push( mod('catalog').then( m => m.cleanupCatalog?.()) .catch(() => {})); // ... etc ... Promise.allSettled(cleanupTasks); // No await — runs in background // User sees content immediately
  • Tab content appears instantly — no 200-800ms blank wait for cleanup
  • Cleanup still runs — just in the background, not blocking the user
  • Fixes the 493ms message handler delay from the debug log
  • Each cleanup task already has ?.() null-guard — safe to fire-and-forget

SB-02 Forced reflow 50ms — style write immediately followed by offsetHeight read High JS (various) ▾

Debug log: Forced reflow while executing JavaScript took 50ms. This is triggered when JavaScript writes a style property (e.g., element.style.height = '500px') and then immediately reads a layout property (e.g., element.offsetHeight). The browser is forced to flush all pending style changes to answer the layout query — this blocks the main thread for 50ms. The Catalog render using repeated DOM appends is the most likely source given the 493ms handler time.

Batch DOM writes using DocumentFragment. Separate all reads from writes.

Causing reflow
// Pattern causing forced reflow: items.forEach(item => { const row = createRow(item); container.appendChild(row); // Each append causes reflow // if followed by any measurement }); // Or: write then read immediately el.style.height = '500px'; const h = el.offsetHeight; // REFLOW!
DocumentFragment batch
// Batch all DOM writes together: const frag = document .createDocumentFragment(); items.forEach(item => { const row = createRow(item); frag.appendChild(row); // No reflow — frag is offscreen }); // Single DOM write at the end: container.appendChild(frag); // One reflow only // Separate reads from writes: // Read FIRST, then write const h = el.offsetHeight; // read el.style.height = (h + 20) + 'px'; // write
