2 Firebase Realtime Database Security Rules
Critical — do this before deploying the customer app. The customer menu has NO authentication wall by design (per the spec's UX decision to minimize friction). Every protection against abuse must live in these rules, not in client-side JavaScript.

Add this under each outlet node (replace pizza with your actual outlet key, repeat for every outlet):

{
  "rules": {
    "pizza": {

      "tables": {
        ".read": true,
        ".write": "auth != null",
        "$tableId": {
          ".validate": "newData.hasChildren(['number','capacity','status','token'])"
        }
      },

      "tableSessions": {
        ".read": true,
        "$sessionId": {
          ".write": "auth != null || (!data.exists() || data.child('status').val() === 'active')",
          "orders": {
            ".write": "auth != null || (data.parent().child('status').val() !== 'closed')"
          },
          "runningTotal": { ".write": true },
          "tax": { ".write": true },
          "grandTotal": { ".write": true },
          "status": {
            ".write": "auth != null || newData.val() === 'billing'"
          }
        }
      },

      "orders": {
        ".read": "auth != null",
        "$orderId": {
          ".write": "auth != null || (!data.exists() && newData.child('source').val() === 'QR')",
          ".validate": "newData.hasChildren(['status','items','total','createdAt'])"
        }
      },

      "tableRequests": {
        ".read": "auth != null",
        ".write": true,
        "$reqId": {
          ".validate": "!data.exists() && newData.hasChildren(['tableId','type','createdAt'])"
        }
      },

      "tableAnalytics": {
        ".read": "auth != null",
        ".write": "auth != null"
      },

      "dineinSettings": {
        ".read": true,
        ".write": "auth != null"
      },

      "categories": { ".read": true },
      "dishes": { ".read": true },
      "settings": {
        "storeName": { ".read": true },
        "customerMenuBgImage": { ".read": true }
      }
    }
  }
}
What these rules enforce:
  • tables: public read (needed to validate the QR token client-side), write requires admin auth only — customers never write directly to a table record
  • tableSessions: public read for the running-bill UI; a session can only be created while it doesn't already exist or is still active, preventing a customer from reopening a closed/billed session
  • orders: a non-authenticated client can only CREATE a new order (never update/delete an existing one), and only if it's tagged source:"QR" — status changes, cancellations, etc. require admin auth
  • tableRequests: create-only, no read — customers can't see other tables' call-waiter history
  • categories / dishes: public read only — menu browsing needs no auth

If your existing rules file already has top-level rules for orders, categories, etc. (very likely, since this is an existing ERP), merge the rules above into your existing structure rather than overwriting it — the new tables, tableSessions, tableRequests, tableAnalytics, and dineinSettings nodes are additions; your existing rules for orders/dishes/categories should be combined with the conditions shown above (use || to OR your existing condition with the new QR-specific one).

4 Deploying the Customer-Facing Menu App

The customer-menu/ folder is a complete, independent static site — it does not get bundled into the Admin panel's build. Deploy it separately (e.g. as a second Firebase Hosting site, a Vercel/Netlify project, or a subfolder of your existing hosting under /menu/).

Step-by-step:
  1. Rename the folder structure as needed for your host (the files are already organized as index.html, css/app.css, js/*.js — most static hosts accept this as-is).
  2. Open Menu/js/firebase.js and paste in the same firebaseConfig values used in Admin/firebase-config.js. This is safe to expose publicly.
  3. Decide your URL shape. The app reads the outlet from the URL path (pathParts[0]) and the token from ?t=. Example: https://menu.roshanipizza.com/pizza/?t=7YH8K2P4X9F6M2A. If you only ever serve one outlet, you can hardcode OUTLET in firebase.js instead.
  4. Set pizza/dineinSettings/qrBaseUrl in Firebase to match wherever you deployed this (e.g. https://menu.roshanipizza.com/pizza/) — tables.js reads this value when generating QR codes, so it always points at the live customer app.
  5. Deploy. Test by adding one table in the admin, opening its QR modal, and scanning it with a phone.
The customer app uses the qrcode.js CDN library (cdnjs.cloudflare.com, MIT licensed) loaded on-demand only when a QR needs generating — it is never loaded for the customer-facing ordering flow itself, keeping that bundle small.
5 Compatibility Notes — Where This Implementation Deviates From the Literal Spec Text
There is exactly one intentional deviation from your architecture document, made to avoid breaking existing code. Everything else follows the spec precisely.
type: "Dine-in" instead of "DineIn"

Your spec's order structure example uses "type":"DineIn" (no hyphen). The existing Admin/js/features/orders.js already defines:

STATUS_SEQUENCES['Dine-in'] = ["Confirmed", "Ready", "Delivered"]

and getStatusOptions() branches on that exact literal string (with the hyphen). If new orders were written with type:"DineIn", they would silently fall through to the default/Online status pipeline — breaking status progression, KDS grouping, and printing for every QR order, with no error thrown anywhere.

Menu/js/order.js therefore writes type:"Dine-in" (matching the pre-existing constant exactly) and adds source:"QR" as a separate field so you can still distinguish a QR-originated dine-in order from one a staff member keys in at the counter via POS. tables.js filters on o.type === 'Dine-in' throughout.

If you'd rather standardize on "DineIn"

This is a 10-minute change if you prefer the no-hyphen spelling project-wide: rename the key in STATUS_SEQUENCES inside orders.js from 'Dine-in' to 'DineIn', update every place that checks o.type === 'Dine-in' (in orders.js itself, and in the new tables.js), and update Menu/js/order.js's payload accordingly. Just make sure all three are changed together in the same deploy.

Everything else matches the spec exactly
  • ✅ No dineinOrders, qrOrders, tableOrders, or any parallel node — orders reuse the single existing /orders path
  • ✅ Tables → Sessions → Orders → Items hierarchy implemented exactly as specified
  • ✅ Session-based billing — multiple orders roll into one tableSessions/{id} record with a running grandTotal
  • ✅ Secure random tokens (crypto.getRandomValues), never a guessable ?table=07 parameter
  • ✅ Table statuses limited to exactly 4 values: free / occupied / billing / disabled — order progress lives entirely in /orders, not duplicated onto the table
  • ✅ Join-or-create session logic — a family scanning the same QR multiple times joins the same session, implemented via a Firebase transaction to avoid race conditions
  • ✅ Customer name/phone collected only at checkout, never before browsing
  • ✅ Exactly 3 Firebase listeners in the admin module (tables, tableSessions, orders) — never one-listener-per-table
  • ✅ DocumentFragment-free but still batched rendering via single innerHTML writes per render pass, not per-row appends
  • ✅ tableAnalytics node populated on session close, exactly as specified
6 Seed dineinSettings Before First Use

Add this node once via the Firebase Console (or a one-time script) under each outlet:

pizza/dineinSettings: {
  "enabled": true,
  "enableSessions": true,
  "enableRequestBill": true,
  "showRunningBill": true,
  "taxPercent": 5,
  "serviceCharge": 0,
  "autoCloseSession": false,
  "qrBaseUrl": "https://menu.roshanipizza.com/pizza/"
}

The customer app reads taxPercent for its bill calculation and qrBaseUrl (via tables.js) for QR generation. If this node is missing entirely, both default to 5% tax and window.location.origin + '/menu/' respectively — the feature still works, just with un-configured defaults.

7 End-to-End Testing Checklist
  1. Open Admin → Tables. Click Add Table, create Table 07 with 4 seats. Confirm it appears in the floor grid as Free.
  2. Click the table card → drawer opens from the right → click View / Print QR → confirm a QR image renders (requires internet for the qrcode.js CDN on first load only).
  3. Copy the QR URL shown under the image. Open it in a phone browser (or a new incognito tab) — confirm the Welcome screen shows "TABLE 07".
  4. Tap Start Ordering → browse categories → tap a dish → customize size/add-ons → Add to Order → confirm the cart badge updates.
  5. Open the cart → confirm subtotal/tax/total are correct → tap Place Order (name/phone optional, leave blank to test the "no contact" path) → confirm it advances to Order Tracking with step 1 ("Order Received") active.
  6. Back in Admin → Tables, confirm: the table card now shows Occupied with a running bill amount; the new order appears in Live Orders (Dine In); the same order appears in the New Orders KDS column.
  7. In the table drawer, click Accept Order-equivalent (the first action button shown matches the order's current status) → confirm the customer's tracking screen updates to "Preparing" within a few seconds (live Firebase listener, no refresh needed).
  8. From the same phone, scan the same QR again (simulating a second guest at the table) → confirm it does NOT create a second session — it should land directly on the Order Tracking screen showing the existing order, not the Welcome screen again.
  9. From the phone, order a second item (tap the "+ Order More Items" button) → place a second order → confirm in Admin that the table drawer's "Current Orders" count is now 2, and the running total is the SUM of both orders, not just the latest one.
  10. From the phone's Call Waiter screen, tap Request Bill → confirm the table card in Admin flips to the Billing status color immediately.
  11. In Admin, click Close Table (Paid) in the drawer → confirm: the table returns to Free, the drawer's order list clears, and (check Firebase directly) pizza/tableAnalytics/{tableId} now has incremented totalOrders and totalRevenue.
  12. Test Bulk QR Print with 2–3 tables created — confirm a new browser tab opens with a print-ready grid of QR cards.
  13. Test the offline path: disable network in devtools, reload the Tables tab — confirm the "Waiting for connection" placeholder appears, then re-enable network and confirm it auto-recovers without a manual refresh.
8 Known Limitations & Intentional Scope Boundaries
  • Discounts/service charge fields exist on the session record (discount, and serviceCharge in dineinSettings) but are not yet wired into the bill calculation UI — order.js and tables.js currently compute totals as subtotal + tax only. Hooking in the existing Discounts module (discounts.js) for dine-in orders is straightforward since it already operates on the same /orders node, but wasn't explicitly requested.
  • Table disable while occupied is blocked at the UI level (no disable button shown for a table with an active session) but not yet enforced by a Security Rule — add a .validate condition on tables/$tableId/status if you want server-side enforcement too.
  • The customer app's "Order Again" flow from the spec's customer flow diagram is implemented as the generic "+ Order More Items" button (returns to the menu) rather than a one-tap repeat of the previous order — repeat-order is a small additive feature on top of cart.js if wanted later.
Resolved during review: each order inside the table drawer now has a working "Open in Orders tab" link. Since the existing orders.js render doesn't currently surface a table number in row text, it switches to the Orders tab and searches by the order's own ID (using the existing #orderSearch input, which already supports "Search ID or name") rather than filtering by table — this avoids touching orders.js's render logic while still getting the admin to the right order in two clicks.
