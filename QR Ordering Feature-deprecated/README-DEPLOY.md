## Verdict: was NOT production ready. Root cause found and fixed below.

The application code (cart → order → session → tracking) was logically
sound — every DOM id referenced in JS exists in HTML, all imports match
real exports, both `ui.js` patches apply byte-for-byte to the live file.
**The blocker was entirely in Firebase Security Rules**, which is exactly
the kind of bug that only surfaces in real end-to-end testing, not a code
read-through.

## 🔴 Confirmed blocking bugs (would have broken the live flow)

### 1. The 5 new RTDB nodes did not exist in the live rules at all
`tables`, `tableSessions`, `tableRequests`, `tableAnalytics`, `dineinSettings`
were **completely absent** from the deployed `database.rules.json`. Firebase
RTDB default-denies any path with no matching rule — so every read/write
to these paths would fail outright, for everyone, including the admin.

### 2. `orders` requires full admin/rider auth — customer app could never create or read its own order
Live rule on `$outletId/orders/$orderId`:
```
.read:  auth != null && (...admin/rider checks only...)
.write: auth != null && (...admin/rider checks only...)
```
There is no unauthenticated branch at all. Since the customer app deliberately
uses **no Firebase Auth** (by design, per `firebase.js`'s own comment), this
would have blocked:
- **Order creation** (`order.js` → `set(push(outletRef('orders')), ...)`)
- **Order tracking**, Screen 5 (`app.js` → `onValue(outletRef('orders/{id}'))`)
- the multi-order **session bill summary** card

### 3. `tables/$tableId` writes had no public path — onboarding would hang forever
`session.js`'s `joinOrCreateSession()` runs a `runTransaction` on
`tables/{id}/currentSession` and an `update()` setting `status:'occupied'`.
With `tables/.write: "auth != null"` and no override, **the very first QR
scan's transaction throws `PERMISSION_DENIED`**, which is never caught
anywhere up the call chain (`joinOrCreateSession → initSession → boot()`).
Result: the loading spinner never clears — the welcome screen never
appears. Same issue blocks **Request Bill** (`status: 'occupied' → 'billing'`).

### 4. `settings/storeName` and `settings/customerMenuBgImage` reads were admin-gated
The welcome screen's brand name + background image reads
(`get(outletRef('settings/storeName'))`, etc.) would also be denied.

## ✅ What was fixed — `database.rules.json`

All fixes were merged into the **real, live rules file** (adapted to its
actual `$outletId` wildcard pattern — the draft doc had assumed a hardcoded
`"pizza"` key, which doesn't match how the live rules are actually structured).

- Added `tables`, `tableSessions`, `tableRequests`, `tableAnalytics`,
  `dineinSettings` nodes under `$outletId`.
- `tables/$tableId` write rule is **narrowly scoped** to the exact 3
  transitions the customer client performs (claim session pointer,
  free→occupied, occupied→billing) — `number`/`capacity`/`token`/`active`
  are immutable from the public side, and there is no path back to
  `free`/`disabled` from the customer. Everything else still requires
  the existing admin/outlet-match condition used elsewhere in the file.
- Extended the **existing** `orders/$orderId` `.read`, `.write`, and
  `.validate` rules (rather than replacing them) with a `source == 'QR'`
  carve-out — admin/rider access is untouched.
- Added public `.read: true` overrides for the two specific `settings`
  leaf fields the welcome screen needs, while the rest of `settings`
  stays admin/rider-gated.

Validated: the merged file parses as valid JSON and every added rule
string was built programmatically (not hand-typed) to keep parentheses
balanced.
