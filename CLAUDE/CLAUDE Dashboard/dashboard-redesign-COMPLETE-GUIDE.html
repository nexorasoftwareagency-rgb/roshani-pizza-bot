📘 Dashboard Redesign — Complete Integration Guide

Desktop (KPI strip + Service Rail) and Mobile (grid-collapse gap fix) in one package. CSS-only — zero HTML edits, zero JS edits, zero build step. Two files to paste, both by appending to the end of an existing file.

1 File Manifest
dashboard-redesign.css

Desktop patch. Appends to Admin/style.css. Merges the 4 KPI tiles into one unified strip, turns Priority Orders into the horizontally-scrolling Service Rail with urgency-colored top edges.

dashboard-redesign-mobile.css

Mobile patch. Appends to Admin/mobile-overrides.css. Closes a pre-existing gap (361–768px) where .dashboard-grid never collapsed to one column — this is what caused the sidebar/main-content overlap on your phone.

Both patches touch only the Dashboard tab. Every rule is scoped with #tab-dashboard or the specific element IDs (#priorityOrderList) — nothing on any of the other 18 tabs changes.

3
Paste the full contents of dashboard-redesign-mobile.css immediately after that closing }. Same note as above — if the file has grown since this delivery, just paste at the very end.
4
Save. Hard-refresh the admin panel on both a desktop browser and a phone (or Chrome DevTools device toolbar) to confirm both files loaded.
Order matters for this one. mobile-overrides.css loads after style.css in index.html (confirmed: line 184 then 186) — that's exactly why the mobile gap fix has to live in this file and not the other one. Don't move it into style.css, or the existing !important mobile rules elsewhere in mobile-overrides.css could re-win the cascade on some viewport widths.
4 What Changed & Why
AreaBeforeAfter
KPI row4 separate .kpi-card-v4 tiles, each with its own orange-tinted shadowOne unified strip, internal dividers, same 4 stats
Priority OrdersWrapping grid, 3-color left accent bar (placed=amber, confirmed=blue, ready=orange)Horizontal scrolling rail, full-width top edge colored by real urgency (placed/pending=red, confirmed=amber, ready=green)
Dashboard grid on phones (361–768px)Never collapsed — sidebar's minmax(280,340px) fought the main column for space, causing visual overlapCollapses to 1 column; main content (Priority + Recent Orders) stacks above the sidebar (Top Spenders, Top Items, Rider Status)

Both KPI/Rail rules are scoped to #tab-dashboard because .kpi-card-v4/.kpi-row are reused on 5 other tabs (POS, Analytics, Payments, Inventory, Rider Analytics) and .priority-card-v4 collides with two pre-existing, conflicting .priority-order-list definitions already in style.css (~line 3031, dead code; ~line 4092, the one that currently wins). ID/tab-scoped selectors win regardless of where those sit — nothing needs to be deleted for this patch to work.

5 Testing Checklist
Desktop
  1. KPI row renders as one bordered strip with thin dividers, not 4 glowing cards. All 4 numbers still populate.
  2. Priority Orders scrolls horizontally. Ticket top edges: red (Placed/Pending), amber (Confirmed), green (Ready).
  3. Click a ticket → order drawer still opens (click delegation in orders.js untouched by this patch, but confirm).
  4. Empty state ("All caught up!") spans the full rail width, not a small pill.
  5. Visit Orders, Analytics, Payments, Inventory, Rider Analytics tabs — KPI rows on all of them look exactly as before (this is the main regression risk given shared class names).
Mobile (real phone, or DevTools device toolbar at 375–428px width)
  1. Dashboard tab: confirm Priority Orders + Recent Order History stack above Top Spenders / Top Items / Rider Status — no overlap between them.
  2. Confirm the rail still scrolls horizontally by touch, with visible momentum (flick and let go — it should coast slightly, not stop dead).
  3. Resize/rotate between ~360px, ~400px, and ~768px — confirm the stacked layout holds at every width in that range (this is the exact gap that was previously broken).
  4. Confirm tablet width (~800–1100px) still shows the existing 2-column tablet layout from mobile-overrides.css's tablet band — this patch doesn't touch that rule.
  5. Confirm Recent Order History still renders as stacked cards (unaffected — this patch doesn't touch .premium-table-v4).
6 Rollback

Both patches are pure additions at the end of their files. To revert, delete everything from the DASHBOARD REDESIGN comment header to the end of each file, and both tabs return to exactly their current behavior — nothing else needs to change back.

7 Flagged, Not Fixed (Out of Scope This Round)
  • Dead CSS: the .priority-order-list block at ~line 3031 of style.css is fully superseded by the one at ~line 4092 — safe to delete whenever you're doing general cleanup.
  • Contrast: .status-badge-v4 (~line 450) uses light text on light backgrounds, the same pattern your audit flagged as NEW-03 on the outlet-status pills.
  • Same mobile gap, other tabs: #tab-categories and #tab-riderAnalytics also use .dashboard-grid and likely have the identical 361–768px collapse gap. Not touched — you scoped this to Dashboard only.
  • Live countdown timers on each ticket (like the KDS elapsed-time badges in Tables) would need a small JS ticking interval, similar to tables.js's _tickKDS() — orders.js's dashboard render is currently a static snapshot per Firebase update. Not included since this pass was scoped to CSS only.
  • Dark mode (audit finding NEW-01) — explicitly deferred to a later pass per your earlier call.
