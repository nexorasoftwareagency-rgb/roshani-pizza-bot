Based on a fresh clone of nexorasoftwareagency-rgb/roshani-pizza-bot (main branch). Every fix below was verified against the live code — not the older draft files in this project's chat history — and the rewritten module was tested with a real dynamic import() to confirm it behaves exactly like it will in the browser.

! The headline finding
The live Settings page currently cannot load at all. Admin/js/features/settings.js has a malformed JSDoc comment around line 18 — a stray * Validates Latitude and Longitude line sitting outside any comment block. This is a genuine JavaScript syntax error. Every dynamic import('./features/settings.js') — which is exactly how both ui.js's tab switcher and main.js's button handlers load this module — rejects with SyntaxError: Unexpected token '*'.

Because switchTab() wraps its data-load in a try/catch that only logs the error, the symptom in production is quiet: clicking Settings just shows an empty/placeholder page with no console-visible crash unless you have devtools open, and Save Settings, Add Fee Slab, Add Offer, image uploads, and the WiFi-password eye toggle for Settings specifically all silently do nothing. This is not a cosmetic issue — no setting has been saveable through the UI since this bug was introduced.

I confirmed this two ways: node --check on the plain .js file gives a false "OK" (Node falls back to a permissive check path for files without an explicit module marker), but a real dynamic import() — the same mechanism the browser uses — throws immediately. The rewritten file in this delivery was verified the same way and loads cleanly.

2 Every bug fixed, and why

In settings.js:

  • Fatal parse error (line ~18–22) — the broken comment block described above. Removed; the JSDoc now reads correctly and the module imports cleanly.
  • Save button permanently loses its icon — the original code did saveBtn.textContent = '⏳ Saving...', which doesn't just change the text, it deletes the <i data-lucide="save"> child element inside the button forever (textContent wipes all child nodes). After the very first save, the button would never show its icon again. Fixed by swapping innerHTML and restoring the original markup afterward — the same safe pattern already used elsewhere in this codebase for the WiFi/Rider password-toggle icons.
  • Defensive null-guards added on a handful of getElementById(...).value reads that would throw if a field were ever missing, without changing behavior when the field is present.

In style.css (3 find-&-replace fixes):

  • .settings-section-card / .section-card-heading — background was white at 3% opacity and heading text was white at 60% opacity. Both are dark-theme values on a site that ships light mode only by default, so every section card in Settings was rendering as a near-invisible box with illegible white-on-white headings. This is most of why the page "feels broken" visually even setting the JS bug aside.
  • .outlet-status-pill (AUTO / OPEN / CLOSED badge) — text colors were light "-400" shades that fail WCAG contrast on the pill's own light background (~2:1). Swapped for darker shades from the same color family, ~7:1+.
  • .border-l-ghost / .border-t-ghost — same dark-theme leftover (white border at 5% opacity, invisible on light backgrounds). These two utilities are used across the whole admin app, not just Settings, so this fix is a safe, app-wide improvement — an invisible border becoming a faint visible one can't cause a visual regression anywhere it's used.

In style.css (new rules — closing real gaps, not redesign preference):

While tracing every class the Settings markup uses (including the ones settings.js builds dynamically for fee-slab rows), about a dozen classes turned out to have zero CSS rule anywhere in the file — .qr-upload-box, .bot-img-upload-box, .kpi-icon-box (only a mobile override existed, no base rule), .location-badge, .form-group-mini, .form-input-small, .btn-icon-danger, and a handful of one-off spacing utilities like .mb-10 / .flex-gap-15 / .br-16. These elements have been rendering with raw browser defaults this whole time. The patch adds minimal, convention-matching rules for all of them.

Mojibake / emoji cleanup: two labels in the live HTML contain corrupted bytes — âœ… Order Confirmed and â­ Feedback / Rating — the classic symptom of a UTF-8 emoji being saved/re-saved through a Latin-1 step somewhere in the edit history. All emoji-as-text headers and labels across the page (✅ 🍕 🚚 🌟 ⭐ 📍 📲 🤖 🍽️) have been replaced with Lucide icons, consistent with how the rest of the admin app (Tables, Discounts, Orders, etc.) already does it — this also makes the labels immune to this exact class of corruption happening again.
3 What's new (not a bug fix — an addition)
  • QR Ordering Base URL field (settingQrBaseUrl, saved to dineinSettings/qrBaseUrl). Today this value can only be set by editing Firebase directly — it's the URL your Tables module (tables.js) reads to build every table's QR code link. It's now editable right inside the Dine-In Charges card, with light URL validation on save.
  • Danger Zone, a new card at the bottom of Settings. Nuclear Refresh is moved here from the sidebar (it used to sit one click away from every other nav item, styled only with orange text — easy to hit by accident). The button keeps the exact same data-action="completeSiteRefresh" attribute, so main.js's existing click handler works without any JS change — this is a pure relocation, not new logic.

5

6
Hard refresh and test using the checklist below.
5 Testing checklist
  1. Open the Admin panel, hard-refresh, click Settings. Confirm the page populates with real data immediately (not blank fields) — this alone confirms the fatal bug is fixed.
  2. Confirm every card is clearly readable: dark text on a light card background everywhere, no white-on-white text.
  3. Change a field (e.g. Store Name), click Save Settings. Confirm the button shows a spinning icon + "Saving...", then returns to its normal "Save Settings" label with its icon still showing afterward.
  4. Reload the tab and confirm the change persisted.
  5. Click Apply Status under Outlet Status — confirm the AUTO/OPEN/CLOSED pill text is clearly legible against its background.
  6. Add a fee slab, add a Today's Offer — confirm both render with proper input borders/spacing, not raw unstyled boxes.
  7. Upload one bot status image (any of the 5) — confirm the 5 upload boxes sit in a proper 2-column grid, not stacked oddly.
  8. Enter a value in the new QR Ordering Base URL field, save, reload — confirm it persists. Open Tables → generate a QR for any table and confirm the link uses this URL.
  9. Scroll to the bottom — confirm Danger Zone renders, and that the sidebar no longer shows "Nuclear Refresh".
  10. Toggle the WiFi password eye icon — confirm it still works (this was already fixed in a prior pass and isn't touched here, just re-verify nothing broke).
Scope discipline: nothing outside the Settings tab was changed. The bot/, menu/, rider/ apps and Firebase data shapes (settings/Store, settings/Delivery, settings/Bot, settings/Display, dineinSettings) were all cross-checked against the live repo and are unchanged — every field this page saves is written under the exact same path and key names those other apps already read.
